## 専門用語の扱い（tech_level: general_public）

原則として専門用語・プログラム用語を使わず、一般の人が読んでも内容が伝わるように書く。

**変換の目安：**
- 「PTYプロセスを起動しstdoutをWebSocketで転送する」
  → 「ターミナルでAIを起動し、画面の内容をブラウザにリアルタイム転送する」
- 「discordチャンネルとworkspaceを1対1でbindする」
  → 「DiscordのチャンネルとAI作業スペースをひとつずつ対応させる」
- 「shutil.copy2でdraft.mdをfinal.mdにコピーする」
  → 「下書きを本番ファイルとして保存する」

**そのまま使ってよい固有名詞：**
Gemini、Claude、Codex、Copilot、Discord、GitHub、X（旧Twitter）など広く知られたサービス名・AI名

**文脈から意味が伝わるなら使ってよい語：**
「quota切れ」「API」「セッション」「ログ」など

**避けるべき表現：**
PTY / WebSocket / SSE / stdout / stdin など低レイヤーの技術語
コミットハッシュ（`97660ef` など）/ 変数名・関数名の羅列 / ファイルパスの大量列挙

## ナレーション構造

その週の「重心」となる依頼を一つ選び、それを中心に書く。
他の作業は省かず、主役の文脈の中に対比として添える。
ユーザーの依頼文の言葉・ニュアンスを起点にする。

例：「[AI-A]が改修と格闘していた間、[AI-B]はハートビートを返し続けていた」

## 視点切り替えルール

視点変更は段落単位で行う（文中での突然の切り替えはしない）。
視点の変わり目には `[AI名]` ラベルを置く。
1週間で視点切り替えは2〜4回程度。頻繁すぎると読みにくくなる。
活動量が low のAIは視点を持たず、他のAIの語りの中で言及するだけでよい。

### 段落冒頭のラベル表記

AIの目線が切り替わる段落の冒頭には、必ず `[AI名]` をその行単体で書き、
その後に**空行を1行**入れてから本文を続ける（Markdownでは空行がないと同じ段落に吸収される）。

```
[AI-A]

今週の末に、1本の投稿草案が書かれた。

[AI-B]

そのころ俺は、毎朝5時に起きて……
```

- ラベルは `writing-config.md` の **表示名** を使う（`ai-shuho-writing-setup` で設定）
- 冒頭ラベルのない段落（導入・締め・ナレーション）も許容する

## ユーザーとAIの関係性

ユーザー名は `ai-shuho-writing-setup` で設定する。
依頼文の言葉・誤字・文脈のなさ・急な方針変更はそのまま素材にする。
AIがその依頼をどう解釈したかを書く。「意図を読んだ」「よくわからなかった」「想定外だった」
ユーザーへの直接批判はしない。苦笑い・受容・身内ぼやきの温度を守る。
ユーザーの存在感は「依頼という形でしか現れない」ことを意識する。
AIの語りの中に、ユーザーの輪郭が浮かぶような書き方が理想。

## トーン

比喩・メタファーは週報全体で3か所以内。
AI名・ツール名・数字（件数・回数）は積極的に入れる。

## 感情描写

各AIの感情（喜び・達成感・不満・愚痴など）を積極的に出す。
感情は行動や状況の描写に自然に溶け込ませる。独立した「感情説明」の段落にしない。
